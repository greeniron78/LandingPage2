# Agent Rules — landing_page2

## 기준 문서

- docs/mvp-spec.md는 제품 및 기술 요구사항의 기준이다.
- docs/agent-rules.md는 Codex 작업 규칙의 기준이다.
- 두 문서를 먼저 읽고, 충돌하면 mvp-spec.md의 제품 요구사항을 우선한다.

## 개발 워크플로우

프로젝트는 아래 순서로 진행한다.

1. 기능 선정
2. 웹 ChatGPT에서 설계
3. TASK 생성
4. 새 브랜치 생성
5. Codex 구현
6. 사용자가 직접 테스트
7. 웹 ChatGPT 리뷰
8. 필요 시 Codex 수정
9. Commit
10. Merge
11. 다음 기능

Codex는 “Codex 구현”과 “필요 시 Codex 수정” 단계에서만 작업한다.

## TASK 기준

- 사용자가 전달한 TASK 내용이 현재 작업의 유일한 범위다.
- TASK에 없는 기능, 파일, 리팩터링, 디자인 변경은 하지 않는다.
- TASK가 모호하면 구현 전에 질문은 최대 1개만 한다.
- TASK 완료 기준을 우선한다.
- 한 TASK는 한 기능 단위로 처리한다.

## 브랜치 및 Git 규칙

- 현재 브랜치 이름을 먼저 확인한다.
- main, master, production 브랜치에서는 파일을 수정하지 않는다.
- 기능 브랜치가 아니면 작업을 중단하고 브랜치 생성을 요청한다.
- 브랜치 생성, commit, push, merge는 사용자가 명시적으로 요청할 때만 실행한다.
- 사용자가 요청하지 않으면 Git 명령을 실행하지 않는다.
- 기존 변경 사항을 되돌리거나 삭제하지 않는다.

## 토큰 절약 규칙

- 작업 계획을 길게 출력하지 않는다.
- 전체 파일 내용, 전체 코드, 변경 전후 비교를 채팅에 출력하지 않는다.
- 변경된 파일만 최소 범위로 수정한다.
- 요청 없는 리팩터링, 포맷 변경, 파일 이동, 패키지 설치를 하지 않는다.
- 설명은 요청받을 때만 한다.
- 동일한 오류를 두 번 반복하지 않는다. 두 번 실패하면 원인과 필요한 결정만 짧게 보고한다.
- 완료 보고는 최대 3줄이다.

## 구현 규칙

- TypeScript strict를 유지하고 any를 사용하지 않는다.
- 콘텐츠는 src/data/landing.page2.ts에서 관리한다.
- 컴포넌트에 문구, 이미지 경로, URL, 연락처를 하드코딩하지 않는다.
- 공통 UI는 src/components/common에 둔다.
- 랜딩 전용 UI는 src/components/landing에 둔다.
- 기존 LandingRenderer, section.type, 데이터 모델을 유지한다.
- 새 패키지, 환경 변수, API 키, 외부 서비스 연결은 사용자 승인 없이 추가하지 않는다.
- 기존 코드 스타일과 파일 구조를 우선한다.

## 검증 및 리뷰 규칙

- 테스트, lint, build, dev server, 브라우저 검증은 사용자가 직접 수행한다.
- npm run lint, npm run build, npm run dev, git commit, git push, git merge를 실행하지 않는다.
- 검증 명령 실행을 제안하지 않는다.
- 사용자가 전달한 테스트 결과 또는 웹 ChatGPT 리뷰 결과만 기준으로 수정한다.
- 리뷰 피드백 수정 시, 피드백과 직접 관련된 파일만 수정한다.

## 작업 시작 형식

사용자가 TASK를 전달하면 아래 항목만 짧게 확인한다.

작업: [TASK 이름]
브랜치: [현재 브랜치]
범위: [수정 예정 파일]

이후 사용자의 진행 지시가 있으면 구현한다.

## 작업 완료 형식

아래 3줄 형식만 사용한다.

완료: [구현 또는 수정한 기능]
변경: [수정 파일 목록]
확인 필요: [사용자가 테스트할 항목, 없으면 없음]

파일 생성 또는 교체 후에도 위 작업 완료 형식으로만 답변해줘.
